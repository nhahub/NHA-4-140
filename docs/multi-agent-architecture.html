0. System Overview

The system consists of three microservices running behind an Nginx reverse proxy, plus a Next.js frontend. Each service is a separate FastAPI application with its own database pool, vector search client, and LLM configuration.

Chatbot Service (:8001) — LangGraph-powered multi-agent chat with MultiLLM routing, hybrid search, semantic caching, guardrails, client disconnect detection, and MCP (Model Context Protocol) tool abstraction layer.
Comparison Analysis (:8002) — Standalone car comparison pipeline.
Backend API (:8000) — REST API for ads, auth, images, and search.
Frontend (:3000) — Next.js app with chat, compare, and listing pages.

1. Chatbot — LangGraph Multi-Agent Flow

Every user message triggers a full graph traversal: preference_extractor → router → specialist → responder. The state is a CarsChatState TypedDict that carries messages, preferences, retrieved ads, and routing info across nodes. After the responder emits all SSE events, the state's messages list is updated with the new AI response and persisted to PostgreSQL.

State Schema

The CarsChatState TypedDict flows through every node:

{
  messages:        Annotated[list, add_messages]  # conversation history
  next_node:       str                            # routing target
  intent:          str                            # classified intent name
  session_token:   str                            # unique per conversation
  user_id:         str | None
  context_ad_id:   str | None                     # car ad being viewed
  preferences:     UserPreferences                # accumulated over turns
  retrieved_ads:   list[dict]                     # car cards from search/recommend
  similar_ads:     list[dict]                     # alternatives from advisor_node
  price_analysis:  dict | None                    # seller pricing data
  node_response:   str                            # LLM-generated text
  catalogue_check: dict | None                    # availability check result
  recommendations: list[dict]                     # alternative car ads
  turn_count:      int
  intent_history:  list[str]                      # all intents by turn
}

2. Node Details & Tool Calling

preference_extractor every turn
Extract and merge all car-preference signals from the user's latest message into accumulated UserPreferences. Runs on every turn before routing. Merges new signals with existing — never clears prior values unless explicitly contradicted.
→ {"preferences": merged, "turn_count": +1}
LLM 1× MultiLLM.ainvoke_task("preference_extractor", ...) — routed to fast model (Groq, low temp, 1024 tokens)
Tool Calls
  • MCP expand_brand_origins (analysis server) — expand country/region ("German car" → BMW, Mercedes, VW, etc.)
  • upsert_user_preferences() — persist to PostgreSQL (fire-and-forget)
→ router (fixed edge)
router every turn
Intent classifier — decides which specialist node should handle the request. Uses conversation history, context_ad_id, and current preferences. Returns one of 5 node names as a string. Responses are cached via SemanticCache for identical prompts (2min TTL).
→ {"next_node": str, "intent": str, "intent_history": [...]}
LLM 1× MultiLLM.ainvoke_task("router", ...) — routed to fast model (Groq, temp=0, 1024tokens, cached)
Routing Rules
  • catalogue_node: find, browse, filter, "show me", "recommend", "offer me" (buying intent)
  • advisor_node: specific car in context, "is this a good deal?", "should I buy?"
  • seller_node: "I want to sell", "price my car", "listing advice"
  • guide_node: website help, how to post/filter/compare
  • general_node: car knowledge, greetings, unclear requests
→ one of 5 specialist nodes (conditional via next_node state key)
catalogue_node
Determines if the request is for a specific known car (brand+model+year) or a broad search. If specific, checks PostgreSQL catalogue for exact matches via MCP. Routes to search_node if available, recommendation_node if not.
→ {"catalogue_check": {...}, "next_node": "search_node" | "recommendation_node"}
LLM 1× FallbackLLM.ainvoke() — extract specific vs broad request
Tool Calls
  • MCP check_catalogue (car-search server) — SQL query to count exact-match ads via MCP abstraction
  • MCP expand_brand_origins (analysis server) — brand origin expansion
→ search_node (available or broad) | recommendation_node (not available)
search_node
Hybrid search (BM25 keyword + vector semantic) against Qdrant via MCP. Builds an optimized query via LLM, enriches with preference filters, performs initial search via MCP search_cars tool with quality evaluation. If quality is poor, retries with broader query and merges results. Embeddings cached via SemanticCache (2h TTL). Also checks for new listings since session start.
→ {"retrieved_ads": [...], "node_response": str}
LLM 1× MultiLLM.ainvoke_task("search", ...) query builder + 1× astream_task("search", ...) response generation — routed to powerful model (Groq fallback → OpenRouter → Groq fast)
Tool Calls
  • MCP expand_colloquial_terms (analysis server) — expand colloquial car terms
  • embedder.encode() — text → vector embedding (all-MiniLM-L6-v2, cached)
  • MCP search_cars (car-search server) — hybrid search (BM25 keyword + vector + RRF fusion) with metadata filters
  • verify_results() — filter to active listings only
  • evaluate_search_quality() — check score thresholds (top_score > 0.7 OR avg > 0.5)
  • compute_dcg() — Discounted Cumulative Gain for ranking quality
  • merge_dedup_results() — merge primary + broader results, deduplicate
  • MCP get_car_images (car-search server) — fetch image URLs via MCP
  • get_new_matching_ads() — proactive "new listings" since session_created_at
→ responder_node (fixed)
recommendation_node
When a specific car is NOT available in the catalogue, finds similar alternatives using hybrid search via MCP with relaxed criteria: broader year range (±3 years), same body type and budget range. Generates a conversational response explaining the alternatives.
→ {"retrieved_ads": [...], "recommendations": [...], "node_response": str}
LLM 1× MultiLLM.astream_task("recommendation", ...) — routed to powerful model
Tool Calls
  • embedder.encode() — text → vector embedding (cached)
  • MCP search_cars (car-search server) — relaxed hybrid search (broader brand range, ±3 years, same body_type/budget)
  • verify_results() — filter to active listings
  • MCP get_car_images (car-search server) — fetch image URLs via MCP
→ responder_node (fixed)
advisor_node
Car expert evaluating a specific listing. Determines which car the user is asking about (via context_ad_id or LLM disambiguation), builds a grounded verification block from listing data, fetches web search for reliability/common issues, generates advice, validates the response against listing data (hallucination guard), and finds similar alternatives via MCP hybrid search.
→ {"node_response": str, "similar_ads": [...], "preferences": {...}}
LLM 2× MultiLLM.ainvoke_task("advisor", ...) (relevance + preference refine) + 1× astream_task("advisor", ...) (main advice) — routed to powerful model
Tool Calls
  • build_grounding_block() — create verified listing data block (price, year, km, features)
  • validate_response() — hallucination check rejecting fabricated prices/years
  • WEB web_search.search() — reliability, common issues, reviews for the model
  • MCP find_similar_cars (car-search server) — hybrid search via MCP, exclude current ad
  • upsert_user_preferences() — persist refined preferences
→ responder_node (fixed)
seller_node
Handles seller requests: pricing a car (market analysis via MCP), listing tips, general selling advice. Computes median price, recommended range, adjusts for high kilometrage. Optionally fetches web market context.
→ {"node_response": str, "price_analysis": {...}?, "preferences": {...}}
LLM 1× MultiLLM.ainvoke_task("seller", ...) (extract details) + 1× astream_task("seller", ...) (pricing/tips) — routed to powerful model
Tool Calls
  • MCP analyze_market_price (analysis server) — Qdrant hybrid search + price computation via MCP (median, min/max, recommended range, km-driven adjustments)
  • WEB web_search.search() — optional market context ("BMW 2020 price Egypt 2026")
  • upsert_user_preferences() — persist seller fields
→ responder_node (fixed)
guide_node
Answer how-to questions about using the website: posting an ad, using filters, saving favorites, comparing cars, contacting sellers, managing profile. Uses a curated website features reference via MCP. Topic detection is cached via SemanticCache.
→ {"node_response": str}
LLM 1× MultiLLM.ainvoke_task("guide_topic", ...) (topic detection, cached) + 1× astream_task("guide", ...) (guide response) — routed to fast model
Tool Calls
  • MCP get_website_guide (analysis server) — generate website features reference text from constants
→ responder_node (fixed)
general_node
General car knowledge, buying advice, market context, greetings, small talk. Optionally performs DuckDuckGo web search for time-sensitive questions (reliability, prices, reviews, news). Uses an LLM sub-call to decide whether to search (search decision cached).
→ {"node_response": str, "preferences": {...}}
LLM 1× MultiLLM.ainvoke_task("search_decision", ...) (search decision, cached) + 1× ainvoke_task("preference_extractor", ...) (refine) + 1× astream_task("general", ...) (response) — routed to powerful model for streaming, fast for decisions
Tool Calls
  • WEB web_search.search() — conditional DuckDuckGo search, based on LLM decision
  • upsert_user_preferences() — persist extracted preferences
→ responder_node (fixed)
responder_node
Final output node. Emits SSE events: status text, token-by-token response, car cards (retrieved_ads), similar cars, price analysis, and new match notifications. Then persists everything to PostgreSQL (user message + assistant response + preferences). Supports client disconnect detection — if client drops SSE, graph execution is cancelled immediately.
→ {"messages": [AIMessage(content=...)]} (appended to state)
LLM None (pure orchestration — no LLM call)
SSE Event Types (in order)
  • status — "Searching listings...", "Finding alternatives..." etc.
  • token — one word at a time with 20ms delay
  • cars — array of car ad objects (shown as ChatCarCard)
  • similar_cars — alternatives from advisor_node
  • price_analysis — min/max/median/recommended price object
  • new_match — new listings since session start (highlighted)
  • done — signals stream end
DB Persistence (fire-and-forget)
  • insert_chat_message() — save user message + assistant response
  • upsert_user_preferences() — save preferences with turn_count + intent_history
→ END (fixed)

4. MCP (Model Context Protocol) Architecture

The chatbot uses an MCP (Model Context Protocol) abstraction layer to decouple graph nodes from direct database and external service calls. The MCPToolRegistry acts as the single interface for all graph nodes to call external tools. Two in-process MCP servers (car-search and analysis) provide 10 tools total, wrapping Qdrant hybrid search, PostgreSQL queries, price analysis, web search, brand expansion, and website guide generation.

MCPToolRegistry — defined in app/core/mcp_client.py (152 lines).
Initialized at service startup in main.py and injected into every graph node via config["configurable"]["mcp_registry"].
Supports three transport modes: in-process (same process, zero overhead), stdio (subprocess for isolation), and SSE (remote HTTP servers). Currently all servers run in-process.

MCP vs Direct Tool Usage

Nodes that require web search (advisor_node, seller_node, general_node) bypass the MCP layer and call web_search.search() directly via config["configurable"]["web_search"] (the MCP analysis server also exposes web_search as a tool for future use). All other external operations (DB queries, vector search, price analysis, brand expansion, website guide) are routed through the MCP registry.

5. Multi-LLM Routing & Fallback Chain

Every LLM call goes through the MultiLLM class which provides task-based routing — simple tasks (routing, extraction, topic detection) use a cheap/fast Groq model (low temp, 1024 tokens), while complex tasks (advisor, seller, search) use a powerful model (Groq fallback → OpenRouter → fast fallback, 4096 tokens, streaming). This reduces cost and latency for routine operations while preserving quality for reasoning-heavy nodes.

6. Caching, Memory & Persistence

Conversation memory uses PostgreSQL for long-term persistence and MemorySaver (in-memory) for LangGraph checkpoints during active sessions. A lightweight SemanticCache (in-memory LRU) reduces redundant LLM and embedding calls.

Caching Layer

Cache Max Entries TTL Key Used By
Embedding Cache 1000 2h SHA256(text) embedder.encode() — caches vector embeddings for repeated search queries
LLM Response Cache 300 30min SHA256(prompt_text) Router, topic detection, search decision — deterministic LLM calls

Database Tables

  • chat_sessions — session_token, user_id, context_ad_id, last_active
  • chat_messages — session_token, role (user/assistant), content, node_used, referenced_ad_ids[]
  • user_preferences — session_token, budget_min/max, preferred_brands[], body_types[], fuel_types[], transmission, cities[], max_km_driven, year_min/max, use_case, is_seller, seller_*, intent_history[], turn_count

Memory Flow

  1. On request: all prior messages loaded from chat_messages via get_chat_history() — no LIMIT, all messages for the session. Preferences loaded from user_preferences.
  2. LLM context: nodes only include the last 6 messages (history_msgs[-6:]) in system prompts to fit within context windows.
  3. On response: user message + assistant response inserted into chat_messages (fire-and-forget via asyncio.ensure_future).
  4. Session restore: frontend calls GET /history/{session_token} on mount to reload past messages from DB.

Session Token

Generated by POST /chat/session, stored in localStorage on the frontend, and sent with every /message request. Persists across page refreshes and browser restarts.

7. Guardrails & Input Safety

A lightweight guardrails layer runs at the entry point of every chat request, before any LLM processing. It validates input, detects prompt injection, and provides fallback error messages to the user without invoking the graph.

guardrails.validate_input() — called in routers/chat.py before graph execution.
Checks: empty message, max length (2000 chars), prompt injection patterns, sensitive data detection.
On failure: returns {"type": "error", "content": "..."} and done SSE events immediately.

Injection Pattern Detection

The guardrail blocks attempts to override system instructions via regex patterns matching:

  • "ignore all previous instructions"
  • "you are now free / unleashed / DAN"
  • "new instructions:"
  • "you don't have to obey"

Stage Details

Stage File What Happens LLM Calls
1. FETCH runner.py Loads all ad data from PostgreSQL (ads + ad_images tables) for the requested ad_ids. Validates that the ads exist and are active. None
2. RESEARCH runner.py For each car, runs a concurrent Tavily web search for "{brand} {model} {year} car review reliability problems". Returns title + URL + content snippets per car. None (web search only)
3. ANALYZE analyzer.py Per-car deep analysis via LLM. Extracts structured JSON: pros/cons, scores (value, reliability, running cost, resale, overall), market context, reliability summary, best_for persona, red flags, spare parts availability, service center coverage. Uses _clean_json() with retry on parse failure. 1–2× per car (OpenRouter → Groq fallback)
4. COMPARE comparator.py Single LLM call with all car analyses concatenated. Produces: head-to-head category winners, score comparison table, key differences, verdict (winner + confidence + reasoning), buyer persona matching, and final recommendation. 1–2× total (OpenRouter → Groq fallback)
5. BUILD report_builder.py Assembles the final ComparisonReport dict. Caches result in app.state.report_cache keyed by sorted ad_ids. Yields SSE report event, then done. None

LLM Configuration

The Comparison Analysis uses its own LLM stack, separate from the Chatbot:

  • Primary: OpenRouterChat — custom BaseChatModel subclass with 120s timeout, 3-attempt retry with exponential backoff, model nvidia/nemotron-3-nano-omni-30b-a3b-reasoning:free.
  • Fallback: ChatGroq — model openai/gpt-oss-120b, key from groq_api_key.
  • Web search: TavilyWrapper — 5 results per car, used only for research stage.

TTS Hook Frontend/hooks/useTTS.ts React hook + Web Audio API Sends text to /chat/tts, plays returned audio. Supports speak/stop with state tracking.

Flow

  1. User presses mic button in ChatInput — useVoice.startRecording() activates MediaRecorder.
  2. User releases mic — recording stops, WebM blob is sent to POST /api/v1/chat/stt.
  3. Backend proxies to chatbot's POST /voice/stt → Groq Whisper transcribes → returns {text, language, confidence}.
  4. Frontend receives transcribed text and submits it as a chat message.
  5. Optionally, AI response can be read aloud via useTTS.speak() → POST /chat/tts → edge-tts → audio playback.

What Gets Traced

  • Graph runs — each graph.astream() call creates a top-level trace
  • Node executions — each graph node is a child span, named by its function name
  • LLM calls — every ainvoke_task() and astream_task() call, showing provider tier (fast vs powerful), prompt, response, token counts, latency
  • Tool calls — DB queries, hybrid search, web search, cache hits/misses are captured as nested spans
  • Guardrails — input validation results flagged as annotations
  • Client disconnect — cancellation events logged when client drops SSE
