Deals Chatbot — Multi-Agent Architecture v2.0.0

LangGraph-powered multi-agent chatbot with hybrid search, MCP tool calling, and LangSmith observability.

1. Service Topology

Chatbot

FastAPI :8001

The multi-agent brain. Hosts the LangGraph graph, all 10 agent nodes, LLM routing (Groq + OpenRouter), embeddings (FastEmbed), hybrid search, and SSE streaming.

Backend

FastAPI :8000

Car ads CRUD, authentication, and SSE proxy. Proxies /message to Chatbot. Manages Supabase PostgreSQL for ads, favorites, users.

Comparison

FastAPI :8002

Dedicated comparison/analysis pipeline. Runs structured comparison, vision analysis, hallucination guard, and report generation. Used by the Frontend's compare feature.

Frontend

Next.js

Next.js SSR application with bilingual (AR/EN) support. SSE consumer for streaming chatbot responses. Containerized via Docker.

Qdrant

Vector DB

Cloud-hosted vector database (cars_ads collection). Hybrid index with BM25 + dense vector (Cosmos). Stores car ad embeddings for semantic + keyword search.

Supabase

PostgreSQL

Persistent storage for chat history (chat_messages), user preferences (user_preferences), sessions (chat_sessions), and car ads (ads).

seller_node nodes/seller_node.py messages[-1], preferences node_response, price_analysis, preferences 1-2x Groq (extraction + streaming)
guide_node nodes/guide_node.py messages[-1] node_response 2x Groq (topic detection + streaming)
general_node nodes/general_node.py messages[-1], preferences node_response, preferences (optional) 2-3x Groq (search decision + streaming + refinement)
responder_node nodes/responder_node.py node_response, retrieved_ads, similar_ads, price_analysis messages (AIMessage) None (pure SSE emission)
Key design principle: Every node receives RunnableConfig so dependencies (LLMs, Qdrant, SSE queue, DB pool) are injected at runtime — no global singletons inside nodes.

4. Tool Calling (MCP)

car-search Server

Tools:

  • search_cars — hybrid (BM25+vector) search with metadata filters
  • get_car_details — full ad from PostgreSQL by UUID
  • find_similar_cars — similar by brand/model/year
  • check_catalogue — availability check for specific make/model
  • get_car_images — image URLs for ad IDs

Dependencies: QdrantSearch, Embedder, DB pool

analysis Server

Tools:

  • analyze_market_price — median/range from similar listings
  • web_search — DuckDuckGo web search for market context
  • get_website_guide — website feature reference for guide_node
  • expand_brand_origins — "German car" → [BMW, Mercedes, VW]
  • expand_colloquial_terms — slang → standard vocabulary

Dependencies: QdrantSearch, Embedder, WebSearch

5. Memory & Context Management

5.1 Layers of Memory

Graph State transient

CarsChatState (TypedDict) holds: messages (add_messages reducer), preferences, retrieved_ads, node_response, intent, turn_count, intent_history. Reset per conversation turn, but previous turns exist in messages history loaded from DB.

Supabase Persistence persistent

chat_messages table: role, content, node_used, referenced_ad_ids. Loaded at start of each turn. user_preferences table: all preference fields + intent_history + turn_count. Updated fire-and-forget per turn.

MemorySaver checkpointer

LangGraph's MemorySaver() checkpointer for thread-level state persistence. Maintains state across graph checkpoints within a single .astream() call. Not used for cross-turn persistence.

Semantic Cache in-memory

Two in-memory caches: embedding_cache (1000 entries, 2h TTL) and llm_response_cache (300 entries, 30min TTL). LRU eviction. Used for cacheable simple tasks like routing decisions.

Intent History persistent

intent_history tracks which specialist node handled each turn (e.g. ["catalogue_node", "advisor_node"]). Stored in user_preferences. Available to all nodes via state["intent_history"].

CostTracker per-request

BaseCallbackHandler that records LLM usage per conversation turn. Tracks prompt/completion tokens, estimated cost, latency per LLM call. Includes set_node_tag() for per-node attribution.

5.2 State Lifecycle Per Turn

┌─ Request arrives
│
├─ 1. Load from Supabase: chat_messages → messages_history,
│                          user_preferences → preferences + intent_history + turn_count
│
├─ 2. Build input_state: { messages, preferences, session_token, user_id, ... }
│
├─ 3. graph.astream() executes nodes sequentially, mutating state:
│       preference_extractor → updates preferences
│       router               → sets next_node + intent
│       catalogue_node       → sets catalogue_check + next_node
│       search_node          → sets retrieved_ads + node_response
│       responder_node       → appends AIMessage, persists to DB
│
├─ 4. After stream completes:
│       CostTracker.summary() → SSE usage_summary event
│
└─ 5. Return SSE stream to caller
Note: The deprecated SessionMemory class (core/memory.py) is NOT used by the LangGraph graph. All persistence goes through Supabase queries directly. The class exists as a legacy reference.

6. LLM Routing & Fallback

MultiLLM Strategy

Task CategoryTasksPrimary ModelFallback Chain
Simple router, preference_extractor, guide_topic, search_decision, catalogue_check Groq (fast): openai/gpt-oss-120b, temp=0 fast model only (no retry)
Complex advisor, seller, search, recommendation, general, comparison Groq (powerful): qwen/qwen3.6-27b, temp=0.3 Groq powerful → OpenRouter → Groq fast 
MultiLLM.ainvoke_task(task_type, messages)
│
├─ task in SIMPLE_TASKS? → Groq fast (no fallback, timeout 25s)
│
├─ task in COMPLEX_TASKS? → try in order:
│     1. Groq powerful    (fallback model, streaming=True)
│     2. OpenRouter       (nemotron-3-nano, fallback API key)
│     3. Groq fast        (last resort)
│
├─ Cache hit? (cacheable tasks: router, guide_topic, search_decision)
│     → return cached LLMResult immediately (30s TTL)
│
└─ All failed? → raise RuntimeError("Service temporarily unavailable")

7.1 Auto-Tracing (Zero Config)

Automatic (no code changes)

  • LangGraph .astream() parent run
  • Each node as a child run (by node name)
  • All ChatGroq/BaseChatModel LLM calls
  • OpenRouterChat._agenerate() (extends BaseChatModel)
  • Errors propagate through callbacks before catch

Manual Instrumentation (@traceable)

  • Embedder.encode() run_type="embedding"
  • QdrantSearch.search() + hybrid_search() run_type="retriever"
  • WebSearch.search() run_type="tool"
  • validate_input() run_type="chain"

7.2 Per-Node Visibility in Dashboard

What you see per node

  • Input state (messages, preferences context)
  • Output state (what the node returned)
  • LLM token usage (prompt + completion)
  • Latency (wall-clock time in ms)
  • Error status with exception details
  • Child spans for embedded/retriever/tool calls

Filtering by conversation

  • Filter by metadata.session_id in LangSmith
  • Filter by metadata.user_id for user-level analysis
  • Filter by metadata.environment ("development" vs "production")
  • Search by tags: "langgraph", "env:development", etc.

# .env (already set)
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=lsv2_pt_...
LANGCHAIN_PROJECT=deals-chatbot-dev    # auto-derived from environment
LANGCHAIN_ENDPOINT=https://api.smith.langchain.com

# If disabled → zero overhead, no errors
LANGCHAIN_TRACING_V2=false                # @traceable becomes no-op
Non-blocking guarantee: When LANGCHAIN_TRACING_V2=false or the API key is missing, LangSmith's internal checks disable all tracing. The @traceable decorator becomes a zero-overhead pass-through. Everything runs exactly as before.

8. Supporting Components

Guardrails core/guardrails.py

Input validation: prompt injection detection (regex patterns), max length check (2000 chars). Output validation: sensitive data (phone numbers) scanning. @traceable for observability.

Hallucination Guard core/hallucination_guard.py

Three-layer protection: (1) verify_results filters inactive ads, (2) build_grounding_block creates a verified data fence, (3) validate_response checks that LLM outputs don't contradict listing data within 5% tolerance.

Semantic Cache core/cache.py

Two in-memory caches (embedding + LLM response) with TTL and LRU eviction. Cacheable tasks: router, guide_topic, search_decision. Embedding cache: 1000 entries, 2h TTL. LLM cache: 300 entries, 30min TTL.

Cost Tracker core/cost_tracker.py

BaseCallbackHandler per conversation turn. Tracks per-run: tokens, cost ($0.15/$0.60 per 1K input/output for Groq), latency. Summary event sent as final SSE event. Per-node tagging via set_node_tag().

FastEmbed core/embedder.py

all-MiniLM-L6-v2 via fastembed. Returns list[float] vectors. Used by search_node, recommendation_node, advisor_node, seller_node for Qdrant queries. Cached in embedding_cache.

Preference Tracker core/preference_tracker.py

Aggregates user preferences across turns. Used by the (deprecated) non-LangGraph orchestrator. In the current graph, preference merging is handled inline by preference_extractor and individual nodes.
